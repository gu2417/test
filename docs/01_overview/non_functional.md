# 비기능 요구사항 (NFR)

`requirements.md` §9 를 설계 관점에서 구체화한다.

## 1. NFR 상세

| ID | 항목 | 목표 | 설계상 반영 |
|----|------|------|-------------|
| NFR-01 | 동시 접속 | ≤100 | `MAX_CLIENTS=128` 정적 세션 배열. accept 시 초과 검사 → `SERVER_FULL` 응답. |
| NFR-02 | 응답 지연 | ≤50ms (로컬) | 브로드캐스트는 논블로킹 `send`, 실패 시 1회 재시도 후 세션 정리. DB 저장은 메시지 전달과 병렬 가능하도록 순서를 `broadcast → insert` 로 함. |
| NFR-03 | 안정성 | 크래시 없음 | 모든 소켓 I/O 는 `SIGPIPE` 무시(`signal(SIGPIPE, SIG_IGN)`), recv 0/에러 시 세션만 정리. 스레드 분리(detach). |
| NFR-04 | 보안 | SHA-256, 평문 전송은 로컬만 | MySQL `SHA2(pass,256)` 저장. 평문 전송은 9장(예외)에서 명시적 리스크로 기록. |
| NFR-05 | 서버 메모리 | ≤100MB | 세션/방/멤버 배열은 **정적 할당**. 메시지 히스토리는 DB 에 위임, 메모리 캐시 없음. |
| NFR-06 | 이식성 | Linux + Windows(MSYS2 GTK4) | 플랫폼 분기는 소켓 헤더(`sys/socket.h` vs `winsock2.h`)와 GTK4 pkg-config 링크 플래그에 국한. |
| NFR-07 | 확장성 | GUI 레이어 교체 가능 | 클라이언트는 `net` 계층과 `ui` 계층을 분리. `net` 은 화면을 몰라야 함. |
| NFR-08 | 영속성 | 재시작 후 유지 | 세션 외 모든 상태는 MySQL 영속. 인메모리 캐시는 **재구성 가능**한 것만. |
| NFR-09 | 스레드 안전 | mutex 보호 | 세션 배열은 `g_sessions_mutex` 로 보호. DB 연결은 **스레드 전용**(공유 금지). |

## 2. 저사양 환경 추가 제약

| 항목 | 제약 | 전략 |
|------|------|------|
| CPU | 저사양 노트북 | busy-wait 금지. 모든 대기는 `recv` blocking 또는 condvar. |
| 메모리 | 클라이언트 상주 ≤20MB | 메시지 뷰 버퍼는 링버퍼(최근 200개)로 제한. |
| I/O | 느린 디스크/터미널 | 매 메시지마다 전체 redraw 금지. 변경 영역만 덧쓰기. |
| 네트워크 | 1Mbps | 패킷 최대 2048B 엄격 준수. 리스트 응답은 페이지네이션. |
| 터미널 | 80x24 보장 | 와이어프레임은 80열 기준. 그보다 작으면 경고 후 계속. |

## 3. 관찰 가능성(최소)

- 서버는 stderr 로 레벨 프리픽스 로그(`[INFO]/[WARN]/[ERROR]`) 출력.
- 구조화된 로그 파일·메트릭 수집은 out-of-scope.

## 4. 테스트 전략(개요, 10 장에서 상세)

- 단위: `chat_program/src/common/utils` · protocol 파서 · password 해시 유틸.
- 통합: 더미 클라이언트로 로그인→방 생성→메시지 왕복 시나리오.
- 부하: 100 동시 접속 mock(NFR-01, 02 검증).
- 회귀: P0 패킷 플로우에 대해 시나리오 스크립트 유지.
