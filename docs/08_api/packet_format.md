# 패킷 포맷

## 1. 프레임

```
<TYPE>|<field1>|<field2>|...|<content>\n
```

- 줄 단위 프레이밍. `\n` (0x0A) 이 레코드 종료자.
- `\r` 은 무시(수신 시 trim). 송신은 `\n` 만 붙임.
- 최대 **2048 byte** (NFR-01). 초과 시 수신 측에서 **연결 drop**.

## 2. 필드 구분자

| 용도 | 구분자 | 예 |
|------|--------|----|
| 필드 | `\|` | `LOGIN\|alice\|hash` |
| 다중값 | `:` | `FRIEND_LIST_RES\|0\|alice:앨리스:1;bob:밥:0` 에서 nick:status 쌍은 `:` 로, 반복은 `;` 로 구분 |
| 리스트 반복 | `;` | `FRIEND_LIST_RES\|0\|alice:앨리스:1;bob:밥:0` |

## 3. content 필드 규칙

- 자유 텍스트(사용자 메시지, 공지 등)는 **항상 마지막** 필드.
- 내부에 `|`, `:`, `;`, `\n` 이 나올 수 있음. 이스케이프 대신 **"마지막 필드까지는 한 번에 취한다"** 규칙으로 처리:
  - 파서는 앞쪽 N개 `|` split 후, 남은 전부를 content 로 간주.
  - `\n` 은 라인 프레이밍 때 이미 제거됨. content 안에 있는 다중행은 클라가 `\\n` 로 치환해 전송(수신 측이 다시 `\n` 으로 복원하지 않고 그대로 표시 — v2.0 은 한 줄 메시지만 허용하므로 실제로는 `\\n` 도 금지).

## 4. 문자 집합

- UTF-8.
- 닉네임/아이디 등 식별자 필드는 ASCII 범위만 허용. content 는 utf8mb4 전체 허용(DB 와 동일).
- 제어문자(0x00~0x08, 0x0B~0x1F, 0x7F)는 **reject**. 단, `\t` (0x09) 는 `\u00A0` 로 치환 후 저장.

## 5. 검증 규칙 (수신 측)

1. `\n` 까지 읽어 한 레코드 완성.
2. 길이 ≤ 2048 검사.
3. 첫 `|` 전까지가 TYPE. 대문자/숫자/`_` 만 허용. 길이 ≤ 32.
4. 각 카테고리별 **필수 필드 수** 이상이면 통과, 부족하면 `SERVER_ERROR`(서버) / drop(클라).
5. 각 필드 길이 상한 검사(패킷 문서별 명시).
6. 필드 안의 불허 구분자 검사:
   - 식별자/숫자 필드에는 `|`, `:`, `;`, 제어문자 절대 불가.
   - content 만 `:`, `;` 허용(단, 구분자 모호성을 주지 않음 — content 가 마지막이라서).

## 6. 타임아웃 · 프레이밍 에러

- 수신 버퍼에 `\n` 이 도착하지 않은 채 **4KB 초과** → drop.
- 30초간 어떤 바이트도 오지 않으면 keep-alive `PING` (클라→서버) 필요. 서버는 90초간 패킷이 없으면 해당 세션 종료.

## 7. 예

```
LOGIN|alice|5e884898...
LOGIN_RES|0|alice|앨리스

CHAT|42|안녕하세요
CHAT_NOTIFY|42|alice|앨리스|3821|안녕하세요

ROOM_LIST_RES|0|1:lobby:로비:30:5:0;2:dev:개발:100:12:1
```
