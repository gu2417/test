# 입력 검증

## 1. 패킷 레벨

| 검사 | 규칙 |
|------|------|
| 최대 길이 | 2048 bytes. 초과 라인은 소켓 close |
| 종단 문자 | `\n` 없이 버퍼가 가득 차면 오류 |
| TYPE 토큰 | 화이트리스트(알려진 TYPE 목록)에 없으면 무시(drop) + 경고 로그 |
| 필드 개수 | 각 패킷별 고정/최소 개수. 부족 시 `RES_BAD_REQUEST` |
| 제어 문자 | `\r`, `\0` 등 포함되면 해당 필드에서 제거하거나 거부 |

## 2. 필드 레벨

| 필드 | 규칙 |
|------|------|
| `id` | `[A-Za-z0-9_]`, 3~20자 |
| `password` | 인쇄 가능 ASCII, 4~30자 |
| `nickname` | 1~20자, 제어문자 금지(한글·이모지 허용) |
| `status_msg` | 0~100자, 제어문자 금지 |
| `room_name` | 1~30자 |
| `topic` | 0~100자 |
| `content` | 1~500자, `\n` 금지(`\n` 은 패킷 종단자) |
| `emoji` (리액션) | 1~20자, 공백 금지 |
| `room_id`, `msg_id` | 양의 정수, 21억 이하 |
| 패킷 내 `:` 사용 | content 필드 내에서는 이스케이프 규칙(다음 절) |

## 3. 구분자 이스케이프

본 프로토콜은 바이너리 이스케이프 대신 **content 는 마지막 필드에 배치** 하여 구분자 문제를 회피한다.
- `ROOM_MSG|<room_id>:<content>` 파싱 시 첫 번째 `:` 만 구분자. 그 이후 바이트는 모두 content.
- 그럼에도 content 내부에 `|` 혹은 `\n` 이 포함되면 **서버가 제거** 한다.

## 4. 거부 응답

검증 실패 시 **간결한** 에러:
```
<ORIG_TYPE>_RES|<code>:<"INVALID">
```
중첩 설명/스택 트레이스 금지.

## 5. 클라이언트 전송 전 검증

- UI 단에서 먼저 길이·문자 검증 → 사용자에게 즉시 피드백.
- **서버는 다시 검증**(이중 방어). 클라이언트 검증을 신뢰하지 않는다.
