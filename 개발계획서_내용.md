## 1장. 프로젝트 개요

### 1-1. 개발의 필요성

현대 대학 생활에서 학과 구성원 간의 원활한 의사소통은 학업 성과와 협업 능력 향상에 직결되는 핵심 요소이다. 카카오톡, 슬랙(Slack), 디스코드(Discord) 등 상용 메신저가 광범위하게 사용되고 있으나, 이 서비스들은 다음과 같은 한계를 가진다.

**상용 메신저의 한계**
- 학과 전용 기능(오픈채팅 강의별 분류, 과제 공지 등) 부재
- 서버가 해외에 위치하여 데이터 주권 문제 존재
- 학습 목적의 시스템 내부 구조 파악 불가
- 특정 플랫폼 종속으로 인한 접근성 제한

**C 언어 GUI 채팅 애플리케이션 개발의 필요성**
- 운영체제 수업에서 배우는 소켓 프로그래밍, 멀티스레딩 개념을 실제 프로젝트에 적용할 수 있는 기회
- C 언어로 GTK4 GUI를 개발함으로써 저수준 시스템 프로그래밍부터 사용자 인터페이스 구현까지의 전 과정 학습
- 서버-클라이언트 분리 구조(TCP 소켓 기반)와 MySQL 영속 저장을 직접 설계하고 구현함으로써 실무 수준의 아키텍처 경험 습득
- 교내 네트워크에서 학과 전용으로 운영 가능한 독립 서버 구축 능력 배양

**기대되는 학습 효과**
네트워크 프로그래밍(TCP/IP 소켓), 멀티스레딩(POSIX pthread), 데이터베이스 연동(MySQL C API), GUI 프레임워크(GTK4) 등 실무에서 요구되는 핵심 기술을 복합적으로 경험할 수 있다.

#### 클라이언트 스레딩 모델

GTK4는 단일 스레드 GUI 프레임워크이므로, 모든 위젯 업데이트는 반드시 GTK 메인 스레드에서 수행해야 한다. 네트워크 수신 스레드에서 UI를 직접 수정하면 경쟁 조건(race condition)이 발생하므로, `g_idle_add()` 함수를 통해 콜백을 GTK 이벤트 루프에 등록하는 방식을 사용한다.

```
[Recv Thread]
  recv() → packet parsing → malloc(UIUpdateData)
     → g_idle_add(ui_callback, data)

[GTK Main Thread]
  GLib event loop → ui_callback(data) 실행
     → GtkTextView에 메시지 추가
     → GtkLabel 업데이트 (읽음, 타이핑 등)
     → free(data)
```

## 2장. 프로젝트 개발 상세 계획

### 2-1. 주요 구성요소 상세 개발 계획

#### 서버 구성요소 상세

서버는 TCP 소켓을 통해 클라이언트 연결을 수락하고, 클라이언트별 전용 스레드에서 패킷을 처리하는 **thread-per-client** 모델로 동작한다.

##### 서버 모듈 목록

| 파일 | 역할 |
|------|------|
| `main.c` | 소켓 생성/바인딩, accept 루프, 클라이언트별 `pthread_create` |
| `config.h` | DB 접속 정보, 포트 번호, `MAX_CLIENTS` 상수 정의 |
| `globals.h / globals.c` | `g_sessions[]` 배열, `g_sessions_mutex` 선언 및 정의 |
| `client_handler.c` | 클라이언트 스레드 본체 — 패킷 수신 루프, 연결 종료 처리 |
| `router.c` | 패킷 타입(`TYPE` 문자열)에 따라 적절한 핸들러 함수로 라우팅 |
| `auth.c` | 회원가입(`REGISTER_REQ`), 로그인(`LOGIN_REQ`), 로그아웃(`LOGOUT_REQ`) 처리 |
| `friend.c` | 친구 추가/수락/거절/목록/삭제/차단 처리 |
| `room.c` | 채팅방 생성/참여/퇴장/초대/공지 처리 |
| `dm.c` | 1:1 DM 전송 및 히스토리 조회 처리 |
| `message.c` | 메시지 저장, 수정, 삭제, 검색, 귓속말, 답장 처리 |
| `broadcast.c` | 채팅방 멤버 전체에게 패킷 브로드캐스트, 알림 전송 |
| `db.c` | MySQL 연결 초기화/종료, Prepared Statement 실행 헬퍼 함수 |

##### 주요 서버 구조체

```c
/* common/types.h — 서버·클라이언트 공용 상수 */
#define MAX_ID_LEN       20
#define MAX_NICK_LEN     20
#define MAX_MSG_LEN     500
#define MAX_ROOM_NAME    30
#define MAX_CLIENTS     128
#define PACKET_BUF_SIZE 2048

/* server/globals.h — 서버 인메모리 세션 */
typedef struct {
    int         fd;                       /* 클라이언트 소켓 fd */
    char        user_id[MAX_ID_LEN + 1];  /* 로그인 ID */
    char        nickname[MAX_NICK_LEN + 1];
    int         online_status;            /* 0=offline, 1=online, 2=busy */
    int         current_room_id;          /* 현재 입장 중인 방 ID (0 = 없음) */
    int         active;                   /* 1 = 유효한 세션 */
    MYSQL      *db;                       /* 스레드 전용 MySQL 연결 */
    pthread_t   tid;
} ClientSession;

extern ClientSession   g_sessions[MAX_CLIENTS];
extern pthread_mutex_t g_sessions_mutex;
```

##### 주요 서버 함수 시그니처

```c
/* client_handler.c */
void *client_handler_thread(void *arg);
/* 역할: 소켓 fd를 인자로 받아 패킷 수신 루프 실행.
         recv() → 패킷 조립 → route_packet() 호출.
         연결 종료 시 세션 정리 후 스레드 종료. */

/* router.c */
void route_packet(ClientSession *sess, const char *type, const char *payload);
/* 역할: type 문자열로 핸들러 함수 테이블에서 해당 함수 조회 후 호출. */

/* auth.c */
int  auth_register(ClientSession *sess, const char *id,
                   const char *pass, const char *nick, const char *status);
int  auth_login(ClientSession *sess, const char *id, const char *pass);
void auth_logout(ClientSession *sess);
/* 반환값: 0 = 성공, 양수 = 오류 코드 */

/* broadcast.c */
void broadcast_to_room(int room_id, const char *packet, int exclude_fd);
/* 역할: g_sessions_mutex 획득 → fd 목록 스냅샷 → mutex 해제
         → 스냅샷 기준으로 send() 수행 (mutex 잠금 중 send 금지) */

void send_notification(int target_fd, const char *type, const char *content);

/* db.c */
MYSQL      *db_connect(void);
void        db_disconnect(MYSQL *db);
MYSQL_STMT *db_prepare(MYSQL *db, const char *sql);
/* sql: ? 플레이스홀더를 사용한 준비된 쿼리 문자열 */

int db_stmt_execute(MYSQL_STMT *stmt, MYSQL_BIND *params, int param_count,
                    MYSQL_BIND *results, int result_count);
/* params: 입력 바인딩 배열, results: 출력 바인딩 배열 (SELECT 시)
   반환: 영향받은 행 수 (성공), -1 (오류) */

void db_stmt_close(MYSQL_STMT *stmt);
```

#### 클라이언트 구성요소 상세

클라이언트는 GTK4 GUI를 기반으로 한 데스크톱 애플리케이션이다. GTK 메인 스레드가 이벤트 루프를 담당하고, 별도의 수신 스레드가 소켓에서 데이터를 읽어 `g_idle_add()`를 통해 UI 업데이트 콜백을 GTK 이벤트 루프에 예약한다.

##### 클라이언트 모듈 목록

| 파일 | 역할 |
|------|------|
| `main.c` | `GtkApplication` 생성, `g_signal_connect("activate", ...)`, GTK 이벤트 루프 시작 |
| `state.h / state.c` | 전역 클라이언트 상태 (`socket_fd`, `my_user_id`, `current_room_id`, `nickname` 등) |
| `net.c / net.h` | 서버 TCP 연결, 수신 스레드 생성, `send_packet()` 함수 (tx_mutex 보호) |
| `app_window.c / app_window.h` | `GtkApplicationWindow` 생성, `GtkStack`으로 화면 전환 관리 |
| `screen_login.c / screen_login.h` | 로그인·회원가입 화면 (`GtkGrid`, `GtkEntry`, `GtkButton`) |
| `screen_main.c / screen_main.h` | 메인 화면 (`GtkPaned`, `GtkNotebook`, `GtkListBox`) |
| `screen_chat.c / screen_chat.h` | 채팅 화면 (`GtkScrolledWindow`, `GtkTextView`, `GtkEntry`) |
| `screen_mypage.c / screen_mypage.h` | 마이페이지 화면 (`GtkGrid`, `GtkWindow` 모달 프로필 수정) |
| `screen_settings.c / screen_settings.h` | 설정 화면 (`GtkDropDown`, `GtkSwitch`) |

#### 메시지 프로토콜 명세

모든 패킷은 `<TYPE>|<PAYLOAD>\n` 형식을 따른다.

| 구성 요소 | 값 | 설명 |
|-----------|-----|------|
| 필드 구분자 | `\|` | TYPE과 PAYLOAD 사이, PAYLOAD 내 필드 구분 |
| 다중값 구분자 | `:` | 단일 필드 내 복수 값 (예: `id:nickname:status`) |
| 리스트 구분자 | `;` | 목록 항목 간 구분 (예: `item1;item2;item3`) |
| 종단 문자 | `\n` | 패킷 끝 |
| 최대 길이 | 2048 bytes | 초과 시 연결 종료 |

> **이스케이프 규칙**: 메시지 내용(`content`), 닉네임, 공지 등 자유 텍스트 필드에 구분자 문자(`|`, `:`, `;`)가 포함될 경우 해당 필드를 마지막 위치에 배치하고 나머지 구분자를 앞에 배치하는 방식으로 파싱한다. 예를 들어 `DM_SEND|<to_id>:<content>` 에서 content는 첫 `:` 이후 모든 문자열로 처리한다.
>
> **비밀번호 처리**: 모든 비밀번호 전송은 평문으로 수행하며, 서버에서 `SHA2(pass, 256)` 해시 후 DB에 저장한다. 네트워크 전송은 로컬/교내 신뢰 망에 한정하여 평문 전송의 위험성을 최소화한다.

#### 데이터베이스 스키마

##### users — 회원 기본 정보

```sql
CREATE TABLE users (
    id            VARCHAR(20)  PRIMARY KEY,               -- 로그인 ID
    password_hash VARCHAR(64)  NOT NULL,                  -- SHA2(pass, 256)
    nickname      VARCHAR(20)  NOT NULL UNIQUE,           -- 닉네임 (중복 불허, 귓속말 대상 식별에 사용)
    status_msg    VARCHAR(100) DEFAULT '',
    online_status TINYINT      DEFAULT 0,                 -- 0=offline, 1=online, 2=busy
    last_seen     DATETIME     DEFAULT NULL,
    created_at    DATETIME     DEFAULT CURRENT_TIMESTAMP
);
```

##### friends — 친구 관계

```sql
CREATE TABLE friends (
    id         INT          AUTO_INCREMENT PRIMARY KEY,
    user_id    VARCHAR(20)  NOT NULL,
    friend_id  VARCHAR(20)  NOT NULL,
    status     TINYINT      DEFAULT 0,                    -- 0=pending, 1=accepted, 2=blocked
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uniq_pair (user_id, friend_id),
    FOREIGN KEY (user_id)   REFERENCES users(id) ON DELETE CASCADE,
    FOREIGN KEY (friend_id) REFERENCES users(id) ON DELETE CASCADE
);
-- 양방향 관계: accepted 시 (A,B), (B,A) 두 레코드 삽입
```

##### rooms — 채팅방

```sql
CREATE TABLE rooms (
    id            INT          AUTO_INCREMENT PRIMARY KEY,
    name          VARCHAR(30)  NOT NULL,
    topic         VARCHAR(100) DEFAULT '',
    password_hash VARCHAR(64)  DEFAULT '',                -- 빈 문자열 = 공개방
    max_users     INT          DEFAULT 30,
    owner_id      VARCHAR(20)  NOT NULL,
    notice        VARCHAR(255) DEFAULT '',                -- 방 공지사항
    is_open       TINYINT      DEFAULT 0,                 -- 0=그룹방, 1=오픈채팅방
    created_at    DATETIME     DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (owner_id) REFERENCES users(id)
);
```

##### room_members — 채팅방 멤버십

```sql
CREATE TABLE room_members (
    room_id   INT          NOT NULL,
    user_id   VARCHAR(20)  NOT NULL,
    open_nick VARCHAR(20)  DEFAULT '',                    -- 오픈채팅 전용 닉네임
    is_admin  TINYINT      DEFAULT 0,                     -- 1=공동방장
    joined_at DATETIME     DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (room_id, user_id),
    FOREIGN KEY (room_id) REFERENCES rooms(id)   ON DELETE CASCADE,
    FOREIGN KEY (user_id) REFERENCES users(id)   ON DELETE CASCADE
);
```

##### messages — 메시지

```sql
CREATE TABLE messages (
    id         INT          AUTO_INCREMENT PRIMARY KEY,
    room_id    INT          DEFAULT NULL,                 -- NULL이면 DM
    from_id    VARCHAR(20)  NOT NULL,
    to_id      VARCHAR(20)  DEFAULT NULL,                 -- DM 수신자
    content    VARCHAR(500) NOT NULL,
    reply_to   INT          DEFAULT NULL,                 -- 답장 원본 msg_id
    msg_type   TINYINT      DEFAULT 0,                    -- 0=일반, 1=시스템, 2=귓속말
    is_deleted TINYINT      DEFAULT 0,
    created_at DATETIME     DEFAULT CURRENT_TIMESTAMP,
    edited_at  DATETIME     DEFAULT NULL,
    FOREIGN KEY (from_id)  REFERENCES users(id),
    FOREIGN KEY (to_id)    REFERENCES users(id),         -- DM 수신자 참조
    FOREIGN KEY (room_id)  REFERENCES rooms(id)  ON DELETE CASCADE,
    FOREIGN KEY (reply_to) REFERENCES messages(id) ON DELETE SET NULL
);
```

##### dm_reads — DM 읽음 상태

```sql
CREATE TABLE dm_reads (
    msg_id    INT          NOT NULL,
    reader_id VARCHAR(20)  NOT NULL,
    read_at   DATETIME     DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (msg_id, reader_id),
    FOREIGN KEY (msg_id)    REFERENCES messages(id) ON DELETE CASCADE,
    FOREIGN KEY (reader_id) REFERENCES users(id)    ON DELETE CASCADE
);
```

##### user_settings — 사용자 커스터마이징

```sql
CREATE TABLE user_settings (
    user_id   VARCHAR(20) PRIMARY KEY,
    theme     VARCHAR(10) DEFAULT 'dark',                 -- 'dark' 또는 'light'
    ts_format TINYINT     DEFAULT 0,                      -- 0=HH:MM, 1=HH:MM:SS
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);
```

##### room_invites — 채팅방 초대 (오프라인 사용자 초대 보관)

```sql
CREATE TABLE room_invites (
    id          INT          AUTO_INCREMENT PRIMARY KEY,
    room_id     INT          NOT NULL,
    inviter_id  VARCHAR(20)  NOT NULL,
    invitee_id  VARCHAR(20)  NOT NULL,
    status      TINYINT      DEFAULT 0,                  -- 0=대기, 1=수락, 2=거절
    created_at  DATETIME     DEFAULT CURRENT_TIMESTAMP,
    UNIQUE KEY uniq_invite (room_id, invitee_id),        -- 중복 초대 방지
    FOREIGN KEY (room_id)   REFERENCES rooms(id)   ON DELETE CASCADE,
    FOREIGN KEY (inviter_id) REFERENCES users(id)  ON DELETE CASCADE,
    FOREIGN KEY (invitee_id) REFERENCES users(id)  ON DELETE CASCADE
);
-- 로그인 시 status=0인 초대를 조회하여 ROOM_INVITE_NOTIFY 전송
```
