### 1-1. 프로젝트 개발의 필요성

- **실시간 의사소통을 위한 경량화된 채팅 시스템 필요**
  - 기존 상용 채팅 애플리케이션(카카오톡, Slack 등)은 GUI 프레임워크가 무겁고 자원 소모가 큼
  - 저사양 PC 환경에서도 구동 가능한 경량 채팅 클라이언트 필요 (상주 메모리 ≤ 20MB 목표)
  - C 언어(C11)로 구현하여 런타임 오버헤드를 최소화한 소켓 기반 통신 구조 구현

- **GTK4 GUI 기반 사용자 인터페이스 필요**
  - 콘솔 TUI 방식에서 탈피, GTK4 GUI 위젯(버튼·입력창·메시지 목록)으로 일반 사용자 접근성 확보
  - 화면 분리 구조(login → main → chat → mypage)로 직관적인 UX 제공
  - GTK4는 Linux와 Windows(MSYS2) 양 환경에서 동일 소스로 빌드 가능

- **보안성 및 데이터 영속성을 갖춘 서버-클라이언트 구조 필요**
  - 비밀번호는 SHA-256 해시(`MySQL SHA2(pass, 256)`)로 단방향 저장 — 평문 보관 금지
  - 모든 DB 쿼리를 Prepared Statement(`?` 플레이스홀더)로 처리하여 SQL Injection 방지
  - MySQL 영속 저장으로 서버 재시작 후에도 모든 유저·방·메시지 데이터 유지 (NFR-08)
  - 서버 비정상 종료 시 크래시 방지: `SIGPIPE` 무시, recv 에러 시 세션만 정리 (NFR-03)

### 1-2. 프로젝트의 목표

- **GTK4 GUI 기반 실시간 채팅 애플리케이션 개발**
  - 서버 1식: TCP 소켓, 멀티스레드(thread-per-client), MySQL 영속 저장
  - 클라이언트: GTK4 GUI 창 기반(로그인/메인/채팅/마이페이지/설정 화면), Linux·Windows 배포

- **카카오톡/Google Chat 핵심 기능 구현 (우선순위 기반)**
  - **P0 (MVP)**: 회원가입·로그인, 그룹 채팅방(생성·참여·퇴장·메시지), 오픈채팅방
  - **P1 (필수)**: 1:1 DM, 친구 관리(추가·수락·차단), 프로필 수정, 읽음 확인
  - **P2 (향상)**: 귓속말, 메시지 삭제·수정, 알림, 방 공지, 멤버 관리
  - **P3 (고급)**: 메시지 검색, 이모티콘, 타이핑 표시, 오픈채팅 익명 닉네임

- **크로스 플랫폼 빌드 지원**
  - 단일 소스 트리에서 `Linux (gcc 9+)` 와 `Windows MinGW (mingw-w64 gcc 8+)` 동시 빌드
  - GTK4 (4.6 이상), pthread, MySQL Connector/C 의존성

- **성능 및 안정성 기준 달성**
  - 동시 접속 ≤ 100 (NFR-01), 응답 지연 ≤ 50ms 로컬 기준 (NFR-02)
  - 서버 상주 메모리 ≤ 100MB (MySQL 제외), 클라이언트 ≤ 20MB (NFR-05)

### 1-3. 프로젝트의 개발 개요

#### 시스템 구성도

```
┌──────────────────────────────────────────────────────────────┐
│              클라이언트 (chat_client, GTK4 GUI)               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────────┐ │
│  │ 로그인   │  │  메인    │  │  채팅    │  │  마이페이지  │ │
│  │ 화면     │  │  화면    │  │  화면    │  │  /설정 화면  │ │
│  └──────────┘  └──────────┘  └──────────┘  └──────────────┘ │
│              GTK 메인 스레드 (이벤트 루프)                    │
│              recv 스레드 ─→ g_idle_add() ─→ UI 갱신          │
└─────────────────────────┬────────────────────────────────────┘
                          │ TCP/IP (TEXT 패킷, \n 종료)
┌─────────────────────────▼────────────────────────────────────┐
│                서버 (chat_server)                             │
│  accept loop ─→ pthread_create ─→ [Handler Thread × N]       │
│  Handler: recv → router.c → 기능 핸들러(auth/room/dm/friend) │
│                           └→ broadcast.c ─→ 대상 클라이언트  │
└─────────────────────────┬────────────────────────────────────┘
                          │ libmysqlclient (Prepared Statement)
┌─────────────────────────▼────────────────────────────────────┐
│                MySQL (chat_db)                                │
│  users │ user_settings │ rooms │ room_members                 │
│  messages │ friends │ dm_reads │ room_invites                 │
└──────────────────────────────────────────────────────────────┘
```

#### 주요 구성 요소

| 구성 요소 | 역할 |
|-----------|------|
| **chat_server** | 단일 프로세스, 스레드 다수. TCP listen + 클라이언트당 핸들러 스레드 생성 + 브로드캐스트 |
| **chat_client** | GTK4 GUI 단일 프로세스. GTK 메인 스레드(이벤트 루프) + recv 스레드 병행 |
| **MySQL** | 유일한 영속 저장소. 유저·방·메시지·친구·설정 전체 저장 |
| **common/** | `protocol.h`(패킷 상수), `types.h`(공유 구조체), `utils.c/h`(타임스탬프 등) |

#### 서버 레이어 구조

```
네트워크 계층   : main.c (accept loop) → client_handler.c (recv 루프)
라우팅 계층    : router.c (패킷 TYPE → 핸들러 함수 테이블)
기능 계층      : auth.c / friend.c / room.c / dm.c / message.c / admin.c
상태/저장 계층 : broadcast.c + g_sessions[] / db.c (MySQL 래퍼) / user_store.c
```

### 1-4. 활용방안 및 기대효과

- **소규모 학습·사내 커뮤니티 의사소통 도구로 즉시 활용**
  - ≤ 100 동시 접속 기준, 학과 또는 소규모 팀 내 채팅 서버로 운영 가능
  - 서버를 학과 서버에 설치하고 학생들은 클라이언트만 설치하여 사용

- **크로스 플랫폼으로 OS 제약 없는 배포**
  - Linux 및 Windows(MinGW) 양 환경에서 동일 소스 빌드 — 별도 Windows 버전 관리 불필요
  - GTK4 기반이므로 향후 macOS(Homebrew GTK4) 지원으로 확장 용이

- **C 언어 소켓 프로그래밍 학습 모델로 활용**
  - TCP 소켓, 멀티스레딩, MySQL C API, GTK4 GUI를 실제 동작하는 애플리케이션에 통합 구현
  - thread-per-client, mutex, g_idle_add() 패턴 등 실무 기법 습득

- **추가 기능 확장의 기반 마련**
  - 네트워크 계층(`net.c`)과 UI 계층(`screen_*.c`) 분리 설계로 GUI 교체·확장 용이 (NFR-07)
  - MySQL → PostgreSQL 등 DB 교체 시 `db.c` 모듈만 수정
  - 추후 TLS 전송 암호화, 파일 전송, 모바일 클라이언트 추가 고려

##### ▪ 메시지 생성기 (패킷 프로토콜)

클라이언트와 서버 간 모든 통신은 **Plain Text 형태의 패킷**으로 수행한다.  
패킷 형식: `TYPE|field1|field2|...|content\n` (최대 2048 byte, UTF-8)

| 메시지 명 | 메시지 목적 | 형태 |
|-----------|-------------|------|
| 회원가입 요청 메시지 | 신규 사용자 계정 생성 요청 | `REGISTER\|id\|password\|nickname` |
| 회원가입 결과 전달 메시지 | 회원가입 처리 결과 전달 | `REGISTER_RES\|code` (`0`=OK, `1`=ID중복) |
| 로그인 요청 메시지 | 사용자 인증 요청 전달 | `LOGIN\|id\|password` |
| 사용자 인증 결과 전달 메시지 | 사용자 인증 결과 전달 | `LOGIN_RES\|code\|id\|nickname` (`0`=OK, `1`=ID불일치, `2`=PW불일치, `3`=이미접속) |
| 로그아웃 요청 메시지 | 세션 종료 및 오프라인 전환 요청 | `LOGOUT` |
| 채팅방 생성 요청 메시지 | 그룹/오픈 채팅방 생성 요청 | `ROOM_CREATE\|name\|is_open\|max_users\|password` |
| 채팅방 생성 결과 메시지 | 채팅방 생성 결과 및 방 ID 전달 | `ROOM_CREATE_RES\|code\|room_id` |
| 채팅방 참여 요청 메시지 | 특정 채팅방 참여 요청 | `ROOM_JOIN\|room_id\|password` |
| 채팅방 참여 결과 메시지 | 참여 결과 및 방 정보 전달 | `ROOM_JOIN_RES\|code\|room_id\|room_name` |
| 채팅방 퇴장 요청 메시지 | 특정 채팅방 퇴장 요청 | `ROOM_LEAVE\|room_id` |
| 그룹 메시지 전송 메시지 | 채팅방 내 메시지 전송 | `CHAT\|room_id\|content` |
| 그룹 메시지 수신 메시지 | 채팅방 메시지 수신자 전달 | `CHAT_NOTIFY\|room_id\|from_id\|nickname\|msg_id\|content` |
| 채팅방 목록 요청 메시지 | 참여 중 또는 공개 채팅방 목록 요청 | `ROOM_LIST_REQ\|type` (`group` 또는 `open`) |
| 채팅방 목록 전달 메시지 | 채팅방 목록 데이터 전달 | `ROOM_LIST_RES\|code\|id:name:cur:max:has_pw;...` |
| DM 전송 메시지 | 1:1 다이렉트 메시지 전송 | `DM_SEND\|to_id\|content` |
| DM 수신 메시지 | 1:1 다이렉트 메시지 수신 전달 | `DM_RECV\|from_id\|nickname\|msg_id\|content\|timestamp` |
| 친구 추가 요청 메시지 | 친구 요청 전송 | `FRIEND_REQUEST\|target_id` |
| 친구 요청 결과 메시지 | 친구 요청 처리 결과 전달 | `FRIEND_REQUEST_RES\|code` (`0`=전송됨, `1`=없는ID, `2`=차단됨, `3`=이미친구) |
| 친구 목록 요청 메시지 | 친구 목록 조회 요청 | `FRIEND_LIST` |
| 친구 목록 전달 메시지 | 친구 목록 및 온라인 상태 전달 | `FRIEND_LIST_RES\|code\|id:nick:online;...` |
| 알림 전달 메시지 | 메시지·친구요청·멘션 알림 전달 | `NOTIFY\|type\|content` |

- 패킷 생성 전용 함수 정의:

```c
/* 패킷 직렬화 함수 (packet.c/h) */
int  packet_build(char *buf, size_t buflen, const char *type, ...);
int  packet_parse(const char *line, char *type, char fields[][256], int max_fields);

/* 예: 로그인 결과 패킷 생성 */
packet_build(buf, sizeof(buf), "LOGIN_RES", "0", session->id, session->nickname, NULL);
/* → "LOGIN_RES|0|alice|앨리스\n" */
```

##### ▪ 사용자 인증 (auth.c/h)

클라이언트의 로그인·회원가입 요청 패킷 내 아이디와 패스워드를 검증하여 세션을 생성한다.

- 서버에서 관리하는 사용자 인증 데이터: MySQL `users` 테이블 (Prepared Statement 전용 접근)

```c
/* 사용자 정보 구조체 (common/types.h) */
typedef struct {
    char  id[21];             /* 사용자 ID (≤20자, ASCII) */
    char  password_hash[65];  /* SHA-256 해시 (MySQL SHA2(pass,256)) */
    char  nickname[21];       /* 닉네임 (≤20자, UNIQUE) */
    char  status_msg[101];    /* 상태 메시지 (≤100자) */
    int   online_status;      /* 0=오프라인, 1=온라인, 2=자리비움 */
    int   socket_fd;          /* 접속 중 소켓 FD (-1=오프라인) */
    int   dnd_mode;           /* 방해금지 모드 (0=OFF, 1=ON) */
} User;
```

- 사용자 인증 결과에 따라 결과값 도출:
  - 결과값 `0` : 인증 성공 → 세션 배열 등록, `online_status=1` 갱신
  - 결과값 `1` : 사용자 ID 불일치
  - 결과값 `2` : 사용자 Password 불일치
  - 결과값 `3` : 이미 접속 중인 계정 (`ALREADY_ONLINE`)

- 사용자 인증 비교 결과 기반 클라이언트 전달용 MSG 생성: `packet_build()` 함수 호출
- 클라이언트 전달용 메시지 생성 후 `broadcast.c`의 단일 전송(`notify_user()`)으로 메시지 전송

##### ▪ 채팅방 관리 (room.c/h)

```c
/* 채팅방 구조체 (common/types.h) */
typedef struct {
    int   room_id;
    char  room_name[51];       /* 방 이름 (≤30자) */
    char  owner_id[21];        /* 방장 ID */
    int   is_open;             /* 0=초대제 그룹방, 1=오픈채팅 */
    int   member_count;        /* 현재 멤버 수 */
    int   member_fds[64];      /* 접속 중 멤버 socket_fd 목록 */
} ChatRoom;
```

- 방 생성 (`ROOM_CREATE`): `INSERT rooms` → `INSERT room_members(is_admin=1)`
- 방 참여 (`ROOM_JOIN`): 정원·비밀번호 검사 → `INSERT room_members` → 히스토리 100개 전송
- 방 퇴장 (`ROOM_LEAVE`): `DELETE room_members` → 시스템 메시지 브로드캐스트
- 방장 퇴장 시: 남은 공동방장 중 최장수자에게 자동 위임, 없으면 방 삭제

##### ▪ 메시지 처리 (message.c/h)

```c
/* 메시지 구조체 (common/types.h) */
typedef struct {
    int   msg_id;
    int   room_id;            /* 0 또는 NULL = DM */
    char  from_id[21];        /* 발신자 ID */
    char  to_id[21];          /* 수신자 ID (DM 전용) */
    char  content[501];       /* 메시지 내용 (≤500자) */
    int   msg_type;           /* 0=일반, 1=시스템, 2=귓속말, 3=/me 액션 */
    int   is_deleted;         /* 삭제 여부 (0=정상, 1=삭제됨) */
    int   read_count;         /* 읽음 카운트 */
} Message;
```

- 메시지 수신 → `INSERT messages` → 접속 중 멤버 fan-out (`broadcast.c`)
- 오프라인 멤버는 DB 히스토리로만 저장, 재접속 시 `ROOM_JOIN` 시 최근 100개 전달

##### ▪ 브로드캐스트 (broadcast.c/h)

- `bcast_room(room_id, packet)`: 방 멤버 전체 `member_fds[]` 스냅샷 복사 후 순차 `send()`

```c
/* 클라이언트 전역 상태 구조체 (state.c/h) */
typedef struct {
    int   sock_fd;            /* 서버 소켓 FD */
    char  my_id[21];          /* 로그인한 사용자 ID */
    char  my_nick[21];        /* 닉네임 */
    int   current_room_id;    /* 현재 활성 채팅방 ID */
    int   screen;             /* 현재 화면 (LOGIN/MAIN/CHAT/MYPAGE/SETTINGS) */
} ClientState;
```

- recv 스레드: 소켓 수신 → `packet_parse()` → `g_idle_add(callback, data)`로 GTK 메인 루프에 UI 갱신 예약
- send: `tx_mutex`로 직렬화하여 thread-safe 전송
- **GTK4 스레드 안전성 원칙**: recv 스레드에서 GTK 위젯 직접 수정 금지 — 반드시 `g_idle_add()` 경유

##### ▪ GTK4 GUI 화면 구성

| 화면 | 모듈 | 주요 위젯 |
|------|------|-----------|
| 로그인/회원가입 | `screen_login.c` | `GtkGrid` + `GtkEntry`(ID/PW) + `GtkButton` |
| 메인 탭 화면 | `screen_main.c` | `GtkNotebook`(친구/채팅방/오픈채팅 탭) + `GtkListBox` |
| 채팅 화면 | `screen_chat.c` | `GtkScrolledWindow` + `GtkListBox`(메시지) + `GtkEntry`(입력) |
| 마이페이지 | `screen_mypage.c` | 프로필 편집 위젯 |
| 설정 | `screen_settings.c` | 테마·타임스탬프 형식 설정 위젯 |

- 화면 전환: `GtkStack`으로 login → main → chat → mypage → settings
- 메시지 링버퍼: `#define CHAT_VIEW_CAP 200` — 최근 200개 초과 시 가장 오래된 것 삭제
